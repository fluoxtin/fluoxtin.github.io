Android 平台上的 ORM 性能测试

Posted on 2021-08-09

Object/Relation Mapping 对象关系映射，是一种程序设计技术，用于实现面向对象编程语言里不同类型系统的资料之间的转换。实际实现相当于创建了一个可在编程语言中使用的“虚拟对象数据库”，简言之，ORM 就是通过实例对象的语法，完成关系型数据库的操作的技术。

为什么会有 ORM

在面向对象设计中，我们的设计原则基于软件工程的基本原则（如耦合，聚合，封装），而关系数据库则是从数学的理论发展而来，两者理论依据相差 甚远，为了解决这种不匹配的现象，对象关系映射技术应运而生。

GreenDao

简单介绍

GreenDao 是面向 Android 的一种轻量级的快捷的 ORM 框架，可以将对象映射到数据库中，底层数据库采用 SQLite DataBase。它针对 Android 进行了高度优化，性能最大化的同时内存消耗极少，此外还支持数据库加密。

App : build.gradle

编写实体类

BaseUser.class

Build project 后同步 User 类，并生成相应的 UserDao 类。

关于注解 ：

  • @Entity
    标识当前类是一个实体类，需要在数据库中建立数据表
  • @Id
    实体所对应的数据库中表的主键
  • @Property
    对应表中字段，如果不设置，则默认 就 java 类属性对应相同的字段
  • @NotNull
    标识属性不能为空
  • @Index
    指定一个属性作为索引中的字段
  • @Transient
    标识当前属性不希望被保存，只是一个暂时的状态
  • @Convert
    用于数据类型的转换

获取并使用 Dao 类

条件查询

通过调用 Dao 的 queryBuild() 方法，我们可以获得一个 QueryBuild 对象，使用以下方法可以进行条件查询：

  • limit()
    限制查找的长度
  • offset()
    设置查找时的偏移量
  • where()
    传入一个 WhereCondition 参数，进行查询，如 UserDao.Properties.FirstName.like(“%tom%”)。
  • or()
    可设置 多个 WhereCondition 参数。
  • orderAsc()
    查询结果以升序排列，默认按 id 进行排列。
  • orderDesc()
    查询结果以降序排列，默认按 id 进行排序。

使用 build() 方法返回一个 Query 对象，使用 Query 对象的 list() 方法可以返回一个 List 对象，也可以使用 listLazy() 方法返回一个 LazyList 对象( LazyList 的实现原理时通过持有数据库 cursor 实现的，在使用完毕后需要调用 close() 方法关闭 cursor)。

底层原理

  • DaoMaster & DaoSession
    这两个类，是在引入 GreenDao 的依赖库，并创建好实体类后，Build 项目自动生成的，于此同时生成的类是根据所写实体类生成的 Dao 类。相关流程如下：
    DaoMaster(Database) -> registerDaoClass() -> 创建 TableStatements，用于数据库 SQL 语句操作，将其封装在 daoConfig 中并添加到 dapConfigMap。
    DaoMaster.newSession -> DaoSession(db, IdentityScopeType, Session, daoConfigMap) -> 通过 DaoConfigMap 得到 DaoConfig，创建实体 Dao 类，例如 BaseUserDao。

  • OpenHelper
    OpenHelper 类是 DaoMaster 中的一个抽象内部类，继承自 DataBaseOpenHelper，内有两个构造方法。重写了父类的 onCreate() 方法，在数据库未创建时调用，在这个方法内部调用了 createAllTable() 方法根据所写的实体类创建相应的表。

  • DevOpenHelper
    先看代码，这是 DaoMaster 中的一个静态内部类，继承自 OpenHelper，内有两个构造方法，参数同上。重写了父类的 onUpgrade() 方法，这个方法在数据库版本号不一样时调用（当前版本需要比原来版本大），删除原有数据库，并重建。
    这里没有对原有数据进行保存，所以在实际应用中，我们应用自己定义 OpenHelper 类，在数据库版本更新时导出数据到新版本库中。

不管时创建数据库还是创建表，都是使用的 SQL 语句实现的，GreenDao 将 SQL 语句拼凑封装起来，所以在实际使用中我们不必接触 SQL 语句，用起来会相对方便。

ObjectBox

简单介绍

ObjectBox 是一种对象数据库，它与其他数据库不同点在于，它没有行，列和数据库。数据的存储采用 mdb 文件进行存储。

使用指南

引入依赖

Project ：build.gradle

App : build.gradle

编写实体类

User.class

写完实体类后，Build project 会生成一个 MyObjectBox 类，一个 User_ 类和一个UserCursor。通过 MyObjectBox 获取数据库（BoxStore）, 通过数据库获取对应的表（Box）, 而后即可以进行 CRUD 操作。

获取并使用 Box

条件查询

通过调用 Box 对象的 query() 方法可以获得一个 QueryBuild 对象，通过调用这个对象的一些方法，可以实现条件查询。

  • contains()
    用于查询某个属性包含一个 value。传入参数 Property (实体类属性，由 User_ 提供)，value (提供的条件)，StringOrder（区别是否大小写敏感，默认不敏感）。
  • equal()
    用于查询某种属性匹配一个 value 的数据。传入参数 Property，value
  • order()
    定义查询结果排序的顺序，传入参数 Property。
  • less()
    查询某种属性小于 value 的数据。传入参数 Property，value。
  • greater()
    查询某种属性大于 value 的数据。传入参数 Property，value。
  • startWith()
    查询属性以 value 开始的数据。传入参数 Property， value。
  • endWith()
    查询属性以 value 结束的数据。传入参数 Property， value。

定制条件查询后使用 build() 方法获得一个 Query 对象，使用这个对象的 find(offset, limit) 方法可以获取查询结果。offset 为偏移量，limit 为加载上限，也可以使用 findLazy() 方法获得一个 LazyList 对象。

底层原理

  • MyObjectBox
    对外提供 builder() 方法返回 BoxStoreBuilder，用于创建数据库。getModel() 返回 Model，这里的 model 是给 Native 层创建数据库用的。同时使用 ModelBuilder 将 Entity 和 EntityInfo 关联起来。
  • BoxStore
    代表着整个数据库，由 BoxStoreBuilder 生成。BoxStore 加载了所有的 Native 库，调用 Native 方法创建数据库，调用 Native 方法依次创建数据库表。创建并管理 Box、Transcation。
  • Box
    用于进行 Entity 的 CURD 操作。这里需要了解 Transcation 和 Cursor。
  • Transaction
    数据库管理系统执行中的一个逻辑单位，ObjectBox 中的 BoxStore 是作用之一是用来创建和管理 Transaction。所有的 Transaction 都是通过 BoxStore 的内部方法 beginTx() 和 beginReadTx() 生成。
  • Cursor
    数据库游标，ObjectBox 会为实体类 User 生成一个 UserCursor 类，而所有针对 User 的 CURD 操作实际都是在这个地方发生的。所有游标的创建都必须调用 Transaction 的 createCursor 方法，拿到游标，就可以调用相关方法，进行 CURD 操作。
  • EntityInfo
    保存了 Entity 的相关信息，如名称，属性等，用于后续的一系列操作。

Realm

简单介绍

Realm是一个跨平台的移动数据库引擎，准确来说，它是专门为移动应用所设计的数据持久化解决方案。Realm并不是对SQLite或者CoreData的简单封装，它建立了自己独特的数据库存储引擎，可以方便、高效的完成数据库的各种操作。

获取并使用 realm 对象

条件查询

使用 Realm 对象的 where() 方法，可以返回一个 RealmQuery 对象，使用这个对象的一些方法可以进行执行一些条件筛选。

  • like()
    用于执行模糊查询，这里的匹配的内容格式为 “*value*“
  • beginsWith() & endsWith()
    模糊查找的一种，查找某种属性值以 value 开始或结束的数据
  • greaterThan() & lessThan() & equalTo()
    用于查询某种属性值大于/小于/等于 value 的数据
  • contains()
    查找某种属性值包含 value 的数据
  • limit()
    约定查找返回数据的上限
  • findAll()
    返回一个 RealmResults 类型的对象，这个对象类似于 LazyList，查看里面的数据的时候都为 null，只是存储了数据在内存中的一个引用，在我们需要的时候才去从内存中读取数据。

底层原理

  • 对象驱动
    Realm 核心理念就是对象驱动，也是 Realm 的核心原则。

  • MVCC
    MVCC(Multiversion Concurrency Control),一种多版本控制技术，和源代码管理算法类似。可将 Realm 内部的模型看作 Git，因为它包括了分支以及原子性提交的概念，无需完全复制所有数据就可以在多个分支上工作。可以执行一个语义化的写时拷贝类型而无需担心另一个线程上的写操作会对其产生影响。事务当中将会完全摒弃外界的影响。

  • 零拷贝
    在大多数 ORM 数据库中，当要访问一个对象的某个属性的时候，会先将请求转换成 SQL 语句，如果还未连接数据库则创建一个数据库连接，然后将 SQL 语句发送给硬盘执行检索，将匹配检索的数据存放到内存中。因为磁盘上的数据存储格式不能在内存中使用，还需要对数据格式反序列化。而 Realm 跳过了整个拷贝流程，首先文件始终是内存映射的，无论文件是否在内存当中，都能够访问到文件的任何内容（Realm 保证了硬盘上的文件格式都是内存可读的）。

MMKV

简单介绍

MMKV 是基于 mmap 内存映射的 key-value 组件，底层序列化/反序列化使用 protobuf 实现，性能高，稳定性强。

获取并使用 MMKV

首先调用 MMKV 的内部方法 initialize()，传入一个 context 对象，既可以实现 MMKV 的初始化，调用 mmkvWithID() 方法可返回一个 mmkv 对象进行 CURD 操作。

底层原理

  • 内存准备
    通过 mmap 内存映射文件，提供一段可供随时写入的内存块，由操作系统负责将内存写入文件，也就是上层不必关心数据是如何写入存储器，写入操作完全由操作系统执行。
  • 数据组织
    数据序列化方面采用 protobuf 协议，在性能上和功能上都有不错的表现。使用的是通用的 Key/Value 组件，key 限定为 String 字符串类型，value 则是通过 protobuf 协议序列化成统一的内存块。
  • 写入优化
    Protobuf 不支持增量更新功能，每次写入必须全量写入，MMKV 新增了增量更新功能，将增量 kv 对象序列化后，直接 append 到内存末尾，这样同一个 Key 会有若干份数据，最新的数据在最后，那么只需要在程序启动第一次打开 MMKV 的时候，不短用后读入的数据替换之前的值，就可以保证数据是最新有效的。
  • 空间增长
    使用 append 增量更新带来的一个新的问题，就是不断 append，文件大小会增长得不可控，例如同一个 Key 不断更新，是可能耗尽几百M甚至上G空间，然而事实上，整个 kv 文件就一个 key，不到 1K 的空间就存的下。所以 MMKV 在性能和空间上做了一个折中，以内存 pagesize 为单位申请空间，在空间用尽之前都是 append 模式，当 append 到文件末尾时，进行文件重整、Key 排重，使用序列化保存排重结果，排重后空间还是不够用的话，将文件扩大一倍，直到够用。
  • 数据有效性
    增加 crc 校验，对无效数据进行甄别。

性能比较

ObjectBox，GreenDao，Realm，MMKV 之间的性能对比。

  • 一次插入 n 个实体。 从表中可以看出，一次插入 N 个实体，数据实体比较少的时候，ObjectBox 的速度是比较快的，但是随着插入数据量级的增加，ObjectBox 会慢于 GreenDao。
  • 一次插入 1 个实体，执行 n 次。 考虑到实际应用中，我们可能不会一次性的插入很多个实体，可能会一次存入一个实体，所以进行了这个测试，一次存入一个实体，然后存 n 次。从表中可以 ObjectBox 的插入速度是比较快的，GreenDao 和 Realm 是次之。（表中没有涉及 MMKV 是因为 MMKV 本不支持一次插入 n 个实体，所以在插入 n 个实体中，是通过一次插入一个实体实现的，需要对比可参加上一个图）

总结

综合测试结果看，ObjectBox 的存取性能不如 GreenDao，并且其存储占用空间较大，对于目前环境来看，GreenDao 占用的优势较大，但是 GreenDao 没有一个合适的数据库迁移工具，当数据库变更时需要自己去维护。对于 Realm，其发布至今已有7年之久，在某些方面也是有它的独到之处，比如说它建立了自己的独特的数据库引擎，可以直接对对象进行读写，而不必拘泥于 SQL 语句。MMKV 可以说是在少量数据的插入删除方面做的极好，只需要将数据写入缓存区，后续的存储操作则由操作系统完成。
